# 🏗️ 改造过程 · 从编辑器脚本到独立前端

> 记录从编辑器脚本改造成 pi RP Web 前端的思路。
> 不一定是最优解，只是当时的选择。

## 背景

最初有两个脚本在某个编辑器的插件里运行。脚本依赖编辑器提供的 API，无法在浏览器中独立使用。目标是将其改造为一个独立的 Web 前端，配合 pi 编码代理使用。

## 改造了什么

### 副视角折叠：从插件脚本改到消息渲染管线

| | 改造前 | 改造后 |
|--|--------|--------|
| 触发时机 | 编辑器保存时触发 | 消息渲染到 DOM 前触发 |
| 修改对象 | 直接改 .md 文件内容 | 只改内存中的 HTML |
| 可逆性 | 不可逆 | 可回看原始消息 |

核心代码在 `rp-web-message-renderer.js`：

```javascript
const perspectiveRegex = /<perspective>[\s\S]*?<toggle_title>([\s\S]*?)<\/toggle_title>[\s\S]*?<content_html>([\s\S]*?)<\/content_html>[\s\S]*?<\/perspective>/g;
```

> **为什么用 `[\s\S]*?` 而不是 `.*?`？**  
> 因为 JavaScript 的 `.` 不匹配换行符，`[\s\S]` 匹配所有字符（包括换行）。

### 状态面板：从读聊天消息改到读 state.json

| | 改造前 | 改造后 |
|--|--------|--------|
| 数据源 | 编辑器 API `getChatMessages()` | 后端 `state.json` 文件 |
| 获取方式 | 直接调用 API | WebSocket 发送命令 → 后端读取 |

后端加了一条命令：

```javascript
case "get_rp_state":
  sendToRP(ws, { type: "rp_state", data: deepClone(state) });
  break;
```

### 变量更新：从 `_.set()` 改到 `update_state` 工具

| | 改造前 | 改造后 |
|--|--------|--------|
| 触发方式 | AI 输出 `_.set()` → 编辑器直接执行 | AI 输出 `<UpdateVariable>` → 后端工具执行 |
| 文件操作 | 直接读写文件 | 通过工具 API 操作 |

## 关键设计决策

| 选择 | 为什么 |
|------|--------|
| RP Web 单独起端口 3012 | 不和 tau-mirror（3001）冲突 |
| 前端文件放子目录 | 避免 pi 误加载为扩展 |
| 正则在前端做 | 不改原始消息，RP 模式可开关 |
| 状态从 state.json 读 | 前后端分离 |
| WebSocket | 比轮询/SSE 更适合双向实时通信 |
| XML 标签 | AI 输出流式生成，XML 比 JSON 更容易解析 |

## 遇到的问题

### 1. pi 把前端 JS 当扩展加载
- **症状**: `window is not defined`
- **原因**: pi 扫描 `extensions/` 目录下所有 `.ts`/`.js` 文件
- **解决**: 前端文件放到 `rp-web/` 子目录

### 2. path.join 在 Windows 上的行为
- **症状**: 页面返回 `Not Found`
- **原因**: `path.join("/dir", "/file.html")` 返回 `/file.html`
- **解决**: 去掉路径开头的 `/`

### 3. 两个扩展抢 session_start
- **症状**: RP Web 服务器没启动
- **解决**: 合并到同一个 `session_start` 里

### 4. RP 模式默认关闭
- **症状**: 打开页面看不到副视角和选项
- **原因**: `rpMode` 默认 `false`
- **解决**: 改为默认 `true`

### 5. 对话历史存全局目录
- **原因**: pi 默认存 `~/.pi/agent/sessions/`
- **解决**: `settings.json` 加 `"sessionDir": ".pi/sessions"`

## 经验总结

1. **前后端分离**：前端只负责渲染，后端只负责数据和工具逻辑
2. **不改原始数据**：渲染时的转换不影响原始消息，保留回看能力
3. **渐进式改造**：每个功能点独立改造，逐个验证
4. **XML 作为中间格式**：AI 原生友好，流式生成无障碍
